1. Introduction

1.1 Purpose

The education system is gradually getting online-based. Following the trend, the system aims to provide education online. The main specialty of the system is that it will monitor the students while they are reading the study materials. The system will also let users create classes, join classes, provide teaching materials and most importantly observe a student's reading activity. The system will allow the teachers to check if students are reading the materials or not with the help of face detection and face tracking. It will also help the teachers to find out which section of the provided teaching materials the students spent most of their time so that the teacher can put more emphasis on that section. All in all, the system enhances the teacher's ability to keep track of the student's reading activity.

1.2 Document Conventions

This SRS follows the IEEE standards for writing Software Requirements Specification. According to that conventions all texts are written in Times New Roman font and all headings are written in bold format.

1.3 Intended Audience and Reading Suggestions

This document is intended for all who are interested in this project including project members, developers, project managers, marketing staff, users, testers, and documentation writers. The rest of the document contains an overall description of the project, external interface requirements, system features, functional and non-functional requirements, use and extended use case diagrams. The readers are advised to read it serially so that they don't miss anything and it is easier to understand.

1.4 Product Scope

The product is for teachers and students to ensure a reliable online education system where the teachers will able to keep an eye on the student through the system. There are many online education systems. But none of them provide these advantages to the teachers so that they can monitor the students while they are reading the study materials. This product will let the teachers check if students are reading the materials or not with the help of face detection and face tracking. It will also let the teachers find out which section of the provided teaching materials, the students spent most of their time so that the teachers can put more emphasis on that section.

2. Overall Description

2.1 Product Perspective

Since the rapid spread of COVID-19, online learning has become an inevitable part and parcel of daily life. As such, this product will provide the teachers with the means to give reading materials to their students and monitor student's reading activity. The product helps the teachers by identifying if the student is reading the contents or not through face recognition. Besides, the product also allows teachers to track the time spent by each student reading the study materials. As a result, the teacher can understand which topics were difficult to grasp and give more emphasis on them.

2.2 Product Functions

The major functions of the product are stated below:

  • Teachers can form classrooms.
  • Students can join classrooms.
  • Teachers can provide reading materials.
  • Students can view the reading materials.
  • Teachers can view the reading info of the student.
  • Both the teacher and the student has to log in or sign up first to perform any of the mentioned functions above.

2.3 User Classes and Characteristics

The system generally will be used by two user classes-namely Student and Teacher.

2.3.1 Student

Students will use this product to read the study materials provided by the teacher. Students will be able to join classrooms under a particular teacher and study the reading materials uploaded by the teacher.

2.3.2 Teacher

Teachers will use this product to create classes and upload study materials. Teachers will also be able to monitor a student’s reading activity. This product is useful for teachers that want to track students reading activity and find out the topics which are hard for the students.

2.4 Operating Environment

This product will work on any device supporting web browsers like Chrome, Mozilla Firefox, Microsoft Edge etc. Moreover, the product will also have an android application for android devices with android version 8 or higher.

2.5 Design and Implementation Constraints

Since both the android application and the web-version will be sharing a database, the wait-times of the users might increase if there is a queue of incoming requests. Besides, a constant internet connection must be ensured to run the product correctly. Moreover, several other constraints may arise while building the product.

2.6 User Documentation

Upon completion of the development stage, a user manual will be provided with the product to help users understand how to use the product.

2.7 Assumptions and Dependencies

The system will depend on a number of APIs to perform various tasks. For federated identity service, the system will use an authentication API. For face detection, face recognition and face tracking, the system will call various APIs to integrate the functions successfully. If any of the APIs are unavailable or don’t provide the necessary support, then the system will fail to implement the features. Besides, the system also requires users to have a fully functional front camera and a stable internet connection to operate without any issues.


3. External Interface Requirements

3.1 User Interfaces

When a user enter the website for the first time, the system will respond with a login page prompting the user to login, sign up or sign in with a federated identity. Upon logging in, the user will be directed to the home page where the user will have the option to join a class as a student or create a class as a teacher. If the user wants to join a class, he/she will have insert a code provided by the teacher. He will also have to upload a photo of himself/herself after joining. If the user wants to form a class, he/she will have to name the classroom and generate a code with the help of the system. Furthermore, the user can view reading info of the students after creating a class. The user interface will be further elaborated in the SDS. 

3.2 Hardware Interfaces

The system is compatible with any device supporting web browsers and that has an active internet connection. Alongside the web-platform the system will most likely have an android application which will support all android devices. Moreover, for the system to allow the teachers to monitor student’s reading activity all devices should have a front camera and allow access to the system.

3.3 Software Interfaces

The system will work on any operating system supporting web browsers since it will be hosted on a web server. Moreover, the system will also have an android application supporting android version 8 and above. The system will use Python along with the Django framework as backend tools. For the frontend, we will use HTML, CSS & Bootstrap. Besides, some APIs will be used for federated identity service, face detection, face recognition and face tracking of the students which will be finalized later.

3.4 Communications Interfaces

The system will use a number of communication services. Both the hosted website and android application will be connected to a common database for data communication. Moreover, the system will be using the HTTPS protocol for secure transmission over the internet. Apart from that, the system will use a number of APIs for providing federated identity, face detection, face recognition and face tracking services.

5. Other Nonfunctional Requirements

5.1 Performance Requirements

Since the system integrates AI, it is important to keep the focus on accuracy and speed for better performance. That's why, the system should use the latest API for detecting face/landmark, recognising face and tracking eyes. In the backend, the system will use Postgre SQL with Django in accordance with the documentation of Django. This will reduce the risks of the system being unstable. Besides, it will ensure reliability and help to achieve the desired performance. If the system loses the connection to the Internet, the user should be informed about any changes as soon as he/she gets back online.

5.2 Safety Requirements

In case of fatal errors in the database, like a disk crash or any other temporary failure, the recuperation technique will reestablish a previous duplicate of the database that was supported up to documented capacity. Then it will reproduce a present status by gathering information from the backed up log, up to the hour of failure.

5.3 Security Requirements

The security of the communication between the system and server will be maintained carefully. Since this is a generic app, a user can be both student and teacher. The shared data between the general user and the instructor will be encrypted. The system will use HTTPS protocol to secure the website. HTTPS protects the integrity of any website. It helps prevent intruders from tampering with the communications between a website and its users' browsers.

5.4 Software Quality Attributes

  • Availability: Anyone, anytime from anywhere can access this product. The system can be used from any web browser. Moreover, it will also have a dedicated android application for android users.
  • Maintainability: The system will make sure to give proper services to the users by fixing bugs and pushing updates if necessary.

5.5 Business Rules

Any developer can avail this product's service through API to their website for a cost. Alternatively, general users also can enjoy this service on a subscription basis.
