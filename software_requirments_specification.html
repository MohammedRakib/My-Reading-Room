1. Introduction

1.1 Purpose

Identify the product whose software requirements are specified in this document, including the revision or release number. Describe the scope of the product that is covered by this SRS, particularly if this SRS describes only part of the system or a single subsystem

1.2 Document Conventions

Describe any standards or typographical conventions that were followed when writing this SRS, such as fonts or highlighting that have special significance. For example, state whether priorities for higher-level requirements are assumed to be inherited by detailed requirements, or whether every requirement statement is to have its own priority.

1.3 Intended Audience and Reading Suggestions

Describe the different types of reader that the document is intended for, such as developers, project managers, marketing staff, users, testers, and documentation writers. Describe what the rest of this SRS contains and how it is organized. Suggest a sequence for reading the document, beginning with the overview sections and proceeding through the sections that are most pertinent to each reader type.

1.4 Product Scope

Provide a short description of the software being specified and its purpose, including relevant benefits, objectives, and goals. Relate the software to corporate goals or business strategies. If a separate vision and scope document is available, refer to it rather than duplicating its contents here.

1.5 References

List any other documents or Web addresses to which this SRS refers. These may include user interface style guides, contracts, standards, system requirements specifications, use case documents, or a vision and scope document. Provide enough information so that the reader could access a copy of each reference, including title, author, version number, date, and source or location.

2. Overall Description

2.1 Product Perspective

Describe the context and origin of the product being specified in this SRS. For example, state whether this product is a follow-on member of a product family, a replacement for certain existing systems, or a new, self-contained product. If the SRS defines a component of a larger system, relate the requirements of the larger system to the functionality of this software and identify interfaces between the two. A simple diagram that shows the major components of the overall system, subsystem interconnections, and external interfaces can be helpful.

2.2 Product Functions

Summarize the major functions the product must perform or must let the user perform. Details will be provided in Section 3, so only a high level summary (such as a bullet list) is needed here. Organize the functions to make them understandable to any reader of the SRS. A picture of the major groups of related requirements and how they relate, such as a top level data flow diagram or object class diagram, is often effective

2.3 User Classes and Characteristics

Identify the various user classes that you anticipate will use this product. User classes may be differentiated based on frequency of use, subset of product functions used, technical expertise, security or privilege levels, educational level, or experience. Describe the pertinent characteristics of each user class. Certain requirements may pertain only to certain user classes. Distinguish the most important user classes for this product from those who are less important to satisfy.

2.4 Operating Environment

Describe any items or issues that will limit the options available to the developers. These might include: corporate or regulatory policies; hardware limitations (timing requirements, memory requirements); interfaces to other applications; specific technologies, tools, and databases to be used; parallel operations; language requirements; communications protocols; security considerations; design conventions or programming standards (for example, if the customer’s organization will be responsible for maintaining the delivered software)

2.5 Design and Implementation Constraints

Describe any items or issues that will limit the options available to the developers. These might include: corporate or regulatory policies; hardware limitations (timing requirements, memory requirements); interfaces to other applications; specific technologies, tools, and databases to be used; parallel operations; language requirements; communications protocols; security considerations; design conventions or programming standards (for example, if the customer’s organization will be responsible for maintaining the delivered software).

2.6 User Documentation

List the user documentation components (such as user manuals, on-line help, and tutorials) that will be delivered along with the software. Identify any known user documentation delivery formats or standards.

2.7 Assumptions and Dependencies

List any assumed factors (as opposed to known facts) that could affect the requirements stated in the SRS. These could include third-party or commercial components that you plan to use, issues around the development or operating environment, or constraints. The project could be affected if these assumptions are incorrect, are not shared, or change. Also identify any dependencies the project has on external factors, such as software components that you intend to reuse from another project, unless they are already documented elsewhere (for example, in the vision and scope document or the project plan).

3. External Interface Requirements

3.1 User Interfaces

Describe the logical characteristics of each interface between the software product and the users. This may include sample screen images, any GUI standards or product family style guides that are to be followed, screen layout constraints, standard buttons and functions (e.g., help) that will appear on every screen, keyboard shortcuts, error message display standards, and so on. Define the software components for which a user interface is needed. Details of the user interface design should be documented in a separate user interface specification.

3.2 Hardware Interfaces

Describe the logical and physical characteristics of each interface between the software product and the hardware components of the system. This may include the supported device types, the nature of the data and control interactions between the software and the hardware, and communication protocols to be used.

3.3 Software Interfaces

Refer to documents that describe detailed application programming interface protocols. Identify data that will be shared across software components. 1. System Features

This section includes all the main features of the system depicted with the help of use case diagram. All the use cases have been expanded and explained in details.


1.1 Log In

Use Case Name:

Log In

Use Case Type:

Secondary

Actor(s):

Teacher, Student

Brief Description:

Actors have to provide a username & password to log in to the system.

Typical Course of Events

Actor Actions

System Responses

1. Actor goes to www.sitename.com

2. System responds with login page.

3. Actor enters username, password.

4. System verifies credentials and redirects to site homepage.

Alternate Course of Events

4.1 If actor enters wrong credentials or keeps any field blank, system prompts the actor to reenter credentials or sign-up first.

1.2 Log Out

Use Case Name:

Log Out

Use Case Type:

Secondary

Actor(s):

Teacher, Student

Brief Description:

After completing the desired task, an actor can log out of the system. This use case can be initiated only if the actor is logged in already.

Typical Course of Events

Actor Actions

System Responses

1. Actor clicks the ‘log out’ button.

2. System redirects to login page.

Alternate Course of Events

N/A

1.3 Sign-Up

Use Case Name:

Sign-Up

Use Case Type:

Secondary

Actor(s):

Teacher, Student

Brief Description:

Actor can create an account by providing his email, username & password. This use case can be initiated if the actor is in login page.

Typical Course of Events

Actor Actions

System Responses

1. The actor clicks on the ‘sign-up’ button.

2. The system responds with sign-up page.

3. The actor enters username, password & email.

4. The system validates the credentials and creates an account for the actor redirecting to the login page.

Alternate Course of Events

4.1 If user enters invalid formats of username, password or email, the system prompts the actor to reenter that particular field again.

4.2 The system also prompts the actor to reenter, if the actor’s input matches with an already existing username or email.

1.4 Sign-In With

Use Case Name:

Sign-In With

Use Case Type:

Secondary

Actor(s):

Teacher, Student

Brief Description:

Actor can sign-in with a federated identity instead of creating an account. This use case can be initiated if the actor is in login page.

Typical Course of Events

Actor Actions

System Responses

1. Actor clicks the ‘sign-in with’ button.

2. System responds with the authenticating API of the federated identity.

3. Actor enters username & password

4. API verifies credentials and allows the system to redirect to the system homepage.

Alternate Course of Events

4.1 If the credentials are incorrect or actor keeps any field blank, the API of the federated identity prompts the actor to reenter credentials.

1.5 Form Class

Use Case Name:

Form Class

Use Case Type:

Primary

Actor(s):

Teacher

Brief Description:

Teacher can form a class by generating a code which can then be provided to the students to join the class. This use case can be initiated if the actor is in homepage.

Typical Course of Events

Actor Actions

System Responses

1. The teacher clicks the ‘form a class’ button.

2. The system responds with a create classroom page.

3. The teacher enters class name and clicks the ‘create classroom’ button

4. The system generates a unique code and creates a classroom with the classroom name provided.

Alternate Course of Events

N/A

1.6 Join Class

Use Case Name:

Join Class

Use Case Type:

Primary

Actor(s):

Student

Brief Description:

Student can join a class by inputting the code given by the teacher via any means which is outside the system. This use case can be initiated if the student is in homepage.

Typical Course of Events

Actor Actions

System Responses

1. The student clicks the ‘join a class’ button.

2. The system responds with a join classroom page.

3. The student enters the classroom code provided by the teacher and clicks the ‘join class’ button

4. The system verifies the code and redirects to the classroom page where the student is prompted to upload his/her passport size photo.

5. The student selects his/her photo from the local directory and clicks the ‘upload’ button.

6. The system uploads the photo and shows that the upload is successful.

Alternate Course of Events

4.1 If the student enters an invalid code, the system prompts the student to reenter the classroom code.

6.1 If the system fails to recognize a face in the photo uploaded, it would prompt the student to upload another photo with a face in it.

1.7 Provide Reading Materials

Use Case Name:

Providing Reading Materials

Use Case Type:

Primary

Actor(s):

Teacher

Brief Description:

After creating classroom, only teacher can provide reading materials by uploading it from his/her local directory to the system. This use case can be initiated if the teacher is in a classroom created by the teacher.

Typical Course of Events

Actor Actions

System Responses

1. Teacher clicks ‘provide reading materials’ button.

2. The system responds with a window allowing the teacher to browse his/her local directory and select any file.

3. Teacher selects his/her desired file and clicks the upload button.

4. The system uploads the file and shows that the upload is successful.

Alternate Course of Events

N/A

1.8 Read Reading Materials

Use Case Name:

Read Reading Materials

Use Case Type:

Primary

Actor(s):

Student

Brief Description:

After joining a classroom, the student can view any reading materials provided by the teacher. This use case can be initiated if the student is in a classroom joined by the student.

Typical Course of Events

Actor Actions

System Responses

1. Student clicks the file provided by the teacher.

2. The system responds by opening the file and allowing the student to view and scroll through the file. At the same time, the system turns on the camera of the device and starts using the face recognition and tracking APIs for recognizing face and tracking student’s eye and reading time on the document.

Alternate Course of Events

2.1 If the access to the camera is denied by the student, the system prompts the student to grant access or he/she can’t view the file.

2.2 If the system fails to detect camera on the device, then it would allow the student to view the file without showing any further prompts.

View Reading Info

Use Case Type:

Secondary

Actor(s):

Teacher

Typical Course of Events

Actor Actions

System Responses

2. System responds with the view reading info page where there is a list of names of all the reading materials uploaded.

3. Teacher clicks the name of a particular reading material to view its reading info.

4. System responds with a student reading info page where there are three options which are: detecting face of students, time spent by the students & graph analysis of time spent.

Alternate Course of Events

N/A

Use Case Name:

Detect Face of Students

Use Case Type:

Primary

Teacher

Typical Course of Events

Actor Actions

System Responses

1. Teacher clicks the ‘detect face of the students’ button.

2. The system responds with a list of all the students which states whether or not their faces while reading study materials matched with the photo provided while joining the class.

1.11 Time Spent by Students

Use Case Name:

Time Spent by Students

Use Case Type:

Primary

Actor(s):

Teacher

Brief Description:

Shows the total time spent by the student reading the study materials. It also shows the page by page reading time of the students. This use case can be initiated if the teacher is in the student reading info page.

Typical Course of Events

Actor Actions

System Responses

1. Teacher clicks the ‘time spent by students’ button.

2. The system responds by showing a list of all the students along with the total time spent by each of them reading the study material.

3. The teacher clicks on a particular student.

4. The system responds with the page by page reading time of the student.

Alternate Course of Events

2.1 If the device used by the student doesn’t have any camera, it would show that no camera was detected by the system.

4.1 Here also, if no camera was detected on the device, then the system would display no camera was detected by the system.
