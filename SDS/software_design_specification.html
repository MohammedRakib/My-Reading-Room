1. Introduction

1.1 Purpose

This software design specification is made with the purpose of outlining the software architecture and design of the ‘Reading Room’ in detail. The document will provide developers insight into meeting the client’s needs efficiently and effectively. Moreover, the document facilitates communication and understanding of the system by providing several views of the system design.

1.2 Scope

The Software design document would demonstrate how the design will accomplish the functional and non-functional requirements captured in the Software Requirement Specification (SRS). The document will provide a framework for the developers by describing the high-level components and architecture, subsystems, interfaces, database design, AI components, and algorithm design. This is achieved through the use of architectural patterns, design patterns, sequence diagrams, class diagrams, relational models, and user interfaces.

1.3 Intended audience

The intended audience for this document is venture administrators and developers. This record is intended to be utilized as a rule for all the advanced needs of its makers. The report contains a few specialized phrasings and outlines, hence the perusers are encouraged to have foreordained information about the subjects secured.

1.4 Reference

Software Requirements Specification (SRS) of 'Reading Room'

As we are implementing federated identity those who will sign-in through federated identity will also be stored in this table

Purpose: One of the important parts of our project is to detect the face of our user when he/she is reading provided reading materials that’s why the system requires a pre-uploaded image that will be stored in this table. This pre-uploaded image and webcam image of the user while reading, reading material will be going through match tests many times.

As there is no existence of FaceImage table without User table that is why FaceImage table is a week entity.

 

 

Table Name: ClassRoom

Purpose: Another part of the system is the user can create or join classes. When a user will create a class he/she will be recognized by the system as the teacher of that class in this table and rest who will join the class using unique class code will be listed as a student in the student list of the class by the system in this table. In short with these two and a few other attributes all classroom objects will be stored in this class. To keep track of created/joined classes by the users.
